🐛 트러블슈팅

📌 문제 1: 질문 도메인 판별 실패

- 문제 배경: 단순 키워드 기준 분류는 일반 대화와 과학 질문을 명확히 구분하지 못해, 엉뚱한 문서가 검색되거나 아예 검색이 누락됨.

- 해결 방법: 사용자 질문을 정규화한 뒤, 과학 키워드 리스트 기반 필터링 외에도 Rule 기반 예외처리 및 fallback 답변 로직을 설계하여 안정성을 높임.

📌 문제 2: 검색 결과가 부적절하거나 누락되는 문제

- 문제 배경: FAISS의 유사도 기준만으로는 질문 의도와 동떨어진 문서가 반환되는 경우가 발생.

- 해결 방법: 검색된 문서의 제목/본문 유사도를 추가 필터로 검토해 상위 3개만 노출하고, 필요 시 검색 결과가 없음을 안내하는 예외 응답 구현.

🌟 팀 성과 및 생성 결과 예시

  • ✔ 실제 질문-응답 흐름을 자연스럽게 연결한 UX 구조 완성
  • ✔ MAP 0.89 / MRR 0.89 기록으로 대회 1위 달성
  • ✔ 프롬프트 구성 및 검색 전략 개선으로 응답 일관성과 정확도 향상

💡 깨달은 점

이번 프로젝트를 통해 단순히 모델의 성능보다도, 질문에 적합한 문서를 얼마나 정확히 검색하고, 이를 바탕으로 자연스럽고 신뢰 가능한 응답을 생성하는 전체 정보 흐름의 설계가 더욱 중요하다는 것을 깨달았습니다. 특히 RAG 시스템에서 핵심은 LLM 자체가 아니라, 질문 분류 → 문서 검색 → 응답 생성으로 이어지는 파이프라인 전반의 구조였습니다.

또한, Streamlit을 활용한 사용자 인터페이스 구현 경험은 프론트엔드 시각화의 중요성을 체감하는 계기가 되었습니다. 결과와 출처 문서를 직관적으로 연결해 보여주는 UI 설계가 부족했던 점은 이후 개선 과제로 남았으며, 사용자 경험을 고려한 정보 표현 방식에 대해 고민해보는 기회가 되었습니다.

마지막으로, 생성된 응답의 품질을 정량적으로 평가하기 위한 지표(MAP, MRR)가 유용했지만, 그 한계도 분명했습니다. 특히 사용자 관점에서 '좋은 응답'이 무엇인지에 대한 기준은 정성적 피드백 없이는 포착하기 어렵다는 점에서, 정량과 정성 평가를 함께 고려한 테스트 설계의 중요성을 느꼈습니다.
